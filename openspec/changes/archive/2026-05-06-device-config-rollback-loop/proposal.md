## 为什么

平台目前已具备基线快照、服务端预检、受控应用-验证执行和审计覆盖，但当配置变更以 `verification_failed` 结束或事后判断有害时，没有受控的方式将设备恢复到之前的良好状态。运维人员必须手工构造恢复载荷，而这恰恰是现有安全闭环要消除的操作路径。通过受控回滚路径补齐这一闭环，是平台在真实变更窗口中可被信任的最后一块缺失拼图。

## 变更内容

- 新增受控回滚工作流，将任意历史成功快照转化为新的变更申请，其目标载荷由服务端从快照派生以恢复该快照状态，并复用现有预检、审批、执行和验证流水线。
- 当应用-验证运行以 `verification_failed` 结束时，自动建议（但不自动执行）一个回滚提案，预填来源变更申请的基线快照为回滚目标。
- 在快照视图为 approver/admin 角色新增"恢复到此快照"入口，入口受设备就绪状态和快照资格校验保护。
- 在变更申请记录、预检响应和验证输出中扩展回滚链路字段（`rollback_of_change_id`、`rollback_target_snapshot_id`、`is_rollback`），使审计和详情视图能够区分回滚执行与正向变更。
- 扩展审计日志，新增回滚专属事件：`change.rollback_proposed`、`change.rollback_executed`、`change.rollback_verified`、`change.rollback_verification_failed`。
- 更新运维控制台，在变更详情和历史视图中展示回滚入口、回滚上下文（来源变更、目标快照、比较摘要），并对不符合资格的角色或设备禁用回滚。
- 所有 NETCONF 流量保持在现有 `edit-config` 边界内——回滚引擎不引入 `rollback`、`copy-config` 或其他非受控写原语。回滚本质上是一次普通的受控变更，其载荷由服务端从已知良好快照派生生成。

## 能力

### 新增能力

- `device-config-rollback`：受控回滚工作流、从基线快照派生回滚载荷、回滚专属预检规则（并发约束、目标快照有效性）、回滚验证和回滚审计语义。

### 修改能力

- `device-config-change-control`：变更申请新增回滚链路字段、回滚执行生命周期和回滚感知预检/审批规则。
- `device-config-snapshots`：快照新增回滚目标资格视图，以及服务端派生路径，可为指定快照生成受控 `edit-config` 载荷而不向调用方暴露原始配置内容。
- `change-preflight-verification`：预检新增回滚模式，校验回滚目标快照身份、同设备进行中变更并发约束，以及作为自动建议恢复时的来源变更链路。
- `audit-logging`：审计日志新增回滚专属事件名称和必要上下文字段（`rollback_of_change_id`、`rollback_target_snapshot_id`、来源验证失败码）。
- `operations-console-frontend`：控制台在变更详情和快照视图新增回滚入口、回滚上下文渲染和角色感知动作控制。

## 影响

- 后端 API：新增回滚提案接口，扩展变更申请提交/直接执行路径以接受回滚上下文，扩展预检接口以接受回滚模式，变更申请详情响应新增回滚链路字段。
- 后端服务：变更申请服务、预检服务、NETCONF 任务执行路径、快照服务（新增服务端载荷派生辅助方法）、审计写入器。
- 存储：变更申请记录新增字段（`rollback_of_change_id`、`rollback_target_snapshot_id`、`is_rollback`）及 Alembic 迁移（含降级支持）。预计不需要新表。
- 前端：变更详情和快照视图新增回滚动作、确认流程和回滚上下文渲染；复用既有预检/风险/比较组件。仅在回滚特性导致 `app/page.tsx` 继续膨胀的地方考虑页面级拆分。
- 文档与测试：OpenSpec 规格文档、后端 README 阶段边界更新、回滚路径的后端服务/API 测试、前端类型/构建检查、基于角色的回滚权限控制验证。
